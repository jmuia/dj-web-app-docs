1. Introduction

This introduction section provides background information for the rest of the document. It briefly describes the project, the client deliverables, the project milestones, and expected document changes.

1.1 Project Overview

PartyAll is a web based application that allows guests at a party to democratically curate a playlist for the party they are attending. The purpose of PartyAll is to solve the issues and inefficiencies behind manually creating a playlist that a plurality of people agree on and to avoid having one person monopolizing the music. The issues and inefficiencies that manually creating a playlist for the party includes, but is not limited to, the fact that not everyone will be there at the start of the party to decide upon the songs, a group of people suggesting songs out loud at the same time will cause confusion and for some choices to go unnoticed by the group at large, and preselecting a song playlist means that the playlist will remain static throughout the party and not change as the tastes of the crowd change.

2.2 Organizational Structure

Each task mentioned in 1.2 is assigned to at least one small group to complete. This initial group is accompanied by another small group to perform software quality assurance on the given deliverable. Each group has a representative, however all major decisions are decided amongst the group. This project utilizes the democratic team organization approach. This means that all major decisions will be decided by the group in charge of the specific task.

3. Managerial Process

This section of the SPMP specifies the management process for this project.

3.1 Management Objectives and Priorities

The objective of this project is to produce a web application that democratically controls music played at a party. The main priorities of this project are to complete it with specific functionality and within the time specified (i.e. end of Spring 2014 term - August 20th, 2014). Budget is not a priority because this project does not require one.

3.2 Assumptions, Dependencies, and Constraints

The assumptions of this project are:
i. Deliverables are completed in groups
ii. Each deliverable is reviewed by SQA groups
iii. Availability of software and equipment

The dependencies of this project are:
i. Learning curve for technology group members are inexperienced in

The constraints of this project are:
i. Time as the project must be completed during school term.
ii. Group members inexperience in chosen technology including hardware and software.

3.3 Risk Management

1. Risk: Too many people using the application at the same time may overload the server
i. Application must be scalable
ii. Application must be portable so it can be moved to a new server if need be

2. Risk: Copyright infringement of songs
i. Web application must only access songs it has permission to use

3. Risk: Host abusing privileges
i. Must implement methods of restricting host or simply allow host to act freely

3.4 Monitoring and Controlling Mechanisms

1. Weekly project meetings on Tuesday and Thursday before, after or during class.
2. Shared repository via Github
3. SQA review of every aspect of project
